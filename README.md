# APEX.BUILD

APEX.BUILD is a cloud development platform with a Go backend and a React/Vite frontend. The current codebase covers authentication, project and file management, AI-assisted builds, live preview, terminal sessions, spend/budget controls, BYOK provider routing, and deployment integrations.

## Repo Layout

- `backend/`: Gin-based API, agent orchestration, preview/execution systems, AI provider integrations, auth, billing, and deploy providers.
- `frontend/`: React 18 + TypeScript app for the builder, IDE, preview, admin, settings, and collaboration surfaces.
- `docs/`: authoritative GitHub-facing documentation.
- `tests/`: backend API tests and Playwright E2E infrastructure.
- `render.yaml`: Render blueprint for production deployment.
- `docker-compose.yml`: local multi-service stack.

## Quick Start

### Prerequisites

- Go `1.23+`
- Node.js `18+`
- PostgreSQL `15+`
- Redis `7+` for the full local stack
- Docker Desktop or Docker Engine for containerized local runs and preview/execution features

### Manual local development

```bash
cp .env.example .env

cd backend
go mod download
go run ./cmd/main.go
```

In a second terminal:

```bash
cd frontend
npm install
npm run dev
```

Default local URLs:

- Frontend: `http://localhost:5180`
- Backend health: `http://localhost:8080/health`
- Backend readiness: `http://localhost:8080/ready`
- Backend API base: `http://localhost:8080/api/v1`

### Docker Compose

```bash
docker compose up --build
```

Seeded admin users are disabled by default in local Docker runs. Set `ADMIN_SEED_PASSWORD` and `SPENCER_SEED_PASSWORD`, or opt into the built-in local-only defaults with `ALLOW_DEFAULT_SEED_PASSWORDS=true`.

Default Docker Compose URLs:

- Frontend: `http://localhost:3000`
- Backend: `http://localhost:8080`
- Adminer: `http://localhost:8081` when the `dev-tools` profile is enabled

## Deployment

Production deployment is defined in [`render.yaml`](render.yaml). The frontend Docker image now reads runtime API/WebSocket settings from `/config.js`, which is generated by [`frontend/docker-entrypoint.sh`](frontend/docker-entrypoint.sh). That keeps container deploys aligned with environment variables instead of requiring rebuilds for every endpoint change.

The backend production image now includes its SQL migrations, and `SECRETS_MASTER_KEY` can be supplied either as a base64-encoded 32-byte key or as a strong raw secret that the backend derives into stable key material at runtime.

If you use [`deploy.sh`](deploy.sh) for a standalone Docker deployment, it now refuses to run without explicit production secrets such as `POSTGRES_PASSWORD`, `REDIS_PASSWORD`, `JWT_SECRET`, and `SECRETS_MASTER_KEY`.

Health endpoints:

- Backend liveness: `/health`
- Backend readiness: `/ready`
- Backend feature readiness summary: `/health/features`
- Frontend health: `/health`
